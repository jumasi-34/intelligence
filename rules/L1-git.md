# L1-git.md (L1 Git 사용 및 커밋 메시지 표준)

이 문서는 프로젝트의 형상 관리 일관성을 유지하고, 개발팀과 AI 에이전트 간의 원활한 소통 및 코드 변경 추적을 위해 **Git 사용 및 커밋 메시지 작성 규칙**을 정의하는 **단일 진실 공급원(SSOT) Git 표준 가이드라인**입니다.

---

## 1. Git 커밋 메시지 3대 대원칙 (Three Golden Rules)

모든 커밋 메시지는 프로젝트 멤버 및 AI 협업 도구가 변경 이력을 즉각적이고 명확하게 파악할 수 있도록 아래의 3대 원칙을 반드시 준수해야 합니다.

1. **[한국어 작성]**: 커밋 메시지의 제목과 본문은 **항상 한국어로 명확하고 정중하게 작성**합니다.
2. **[태그 필수 포함]**: 커밋 메시지 시작 부분에는 변경 사항의 성격을 나타내는 **대괄호 태그(`[TAG]`)를 항상 포함**해야 합니다.
3. **[의미 있는 커밋 단위]**: 하나의 커밋은 되도록 하나의 논리적 변경 사항(원자적 작업 단위)만 포함하도록 분할하여 수행합니다.

---

## 2. 커밋 메시지 태그 명세 (Commit Message Tags)

커밋 메시지 헤더(첫 줄)의 맨 앞부분에는 아래 정의된 표준 태그 중 작업 성격에 가장 정확히 부합하는 하나를 선택하여 기재합니다.

| 태그 | 대상 작업 유형 | 구체적인 사용 예시 |
| :--- | :--- | :--- |
| **`[FEAT]`** | 신규 기능 개발 및 추가 | Streamlit 신규 페이지 추가, 신규 필터 컴포넌트 구현 등 |
| **`[FIX]`** | 버그, 오류, 예외 상황 수정 | NullPointerException 방지 로직, DB 커넥션 누수 수정 등 |
| **`[REFACTOR]`**| 비즈니스 로직 수정 없는 구조 개선 | 전처리 코드 체이닝 최적화, 중복 함수 공통화 등 |
| **`[STYLE]`** | 코드 작동에 무관한 스타일/포맷팅 변경 | 자잘한 띄어쓰기 교정, 세미콜론 추가, linter 경고 해결 등 |
| **`[DOCS]`** | 문서 작성 및 수정 | `README.md` 작성, API 가이드라인 추가 등 |
| **`[RULE]`** | 규칙 제정 및 에이전트 행동 가이드 변경 | `GEMINI.md` 수정, `intelligence/rules/` 문서 제정 및 변경 |
| **`[TEST]`** | 테스트 하네스 및 검증 코드 추가 | `tests/` 폴더 내 신규 독립 테스트 작성 및 Mock 수정 등 |
| **`[CHORE]`** | 빌드 설정, 의존성 패키지 관리, 단순 환경 설정 | `requirements.txt` 패키지 추가, `.gitignore` 설정 수정 등 |

## 3. 커밋 메시지 상세 포맷 표준 (Commit Format Standard)

### ① 메시지 구조
커밋 메시지는 가급적 **제목(Header)**과 **본문(Body, 생략 가능)**의 2단계 구조를 갖추는 것을 권장합니다.

```text
[TAG] 요약 제목 (50자 이내, 온점 생략, 명사형 또는 종결어미 사용)

- 변경이 필요한 핵심 배경 및 이유 설명 (본문은 빈 줄을 하나 두고 시작)
- 무엇을 어떻게 변경했는지 핵심 사항 위주로 요약 기록
- 기존 기능에 미치는 부작용(Side Effect)이 있다면 명시
```

### ② 올바른 예시와 잘못된 예시 (Do's & Don'ts)

#### 올바른 예시 (Do's)
> **예시 1 (기능 개발)**
> ```text
> [FEAT] 대시보드 내 공장별 품질 지표 필터링 멀티셀렉트 추가
> 
> - 기존 단일 선택 방식에서 여러 공장을 다중 선택할 수 있도록 개선
> - app/core/params/parameters.py 내 FilterParams 필드 타입 변경
> ```

> **예시 2 (버그 수정)**
> ```text
> [FIX] Databricks SQL 쿼리 빌더 내 날짜 포맷 에러 수정
> 
> - ISO-8601 포맷 문자열이 쿼리 필터링 시 파싱되지 않던 오류 해결
> - SQLConverter 클래스 내 날짜 변환 정규식 보완
> ```

> **예시 3 (규칙 제정 및 갱신)**
> ```text
> [RULE] Git 커밋 규정 가이드라인 신설 및 GEMINI.md 동기화
> ```

#### 잘못된 예시 (Don'ts)
* [안티패턴] `Update page`
  - *이유: 영어 사용 금지 원칙 위배, 태그 누락, 변경 내용을 구체적으로 알 수 없음.*
* [안티패턴] `[FIX] fixed error`
  - *이유: 영어 사용 금지 원칙 위배, 어떤 에러인지 내용 설명 미흡.*
* [안티패턴] `코드 수정함`
  - *이유: 태그 누락 및 구체성 결여.*
* [안티패턴] `[FEAT] 버그 고침`
  - *이유: 버그 수정은 `[FEAT]`이 아닌 `[FIX]` 태그를 사용해야 함.*

## 4. AI 에이전트(Antigravity)를 위한 강제 준수 사항

1. **커밋 명령 자동 제안 시 규칙 확인**: 에이전트가 `run_command` 도구를 사용하여 코드 변경사항`git commit`할 때는 **반드시 이 규칙에 맞춰 커밋 메시지를 동적으로 자동 조립**해야 합니다.
2. **한국어 엄수**: AI가 임의로 영문 메시지를 작성하여 커밋해서는 안 되며, 사용자와 상의되지 않은 사소한 변경 사항이라도 커밋은 반드시 한국어로 작성합니다.
3. **Safety Lock과의 연계**: 프로덕션 코드를 수정하여 커밋할 때는 반드시 사용자의 승인을 구한 후, 승인 여부를 본문이나 요약에 반영하는 것을 권장합니다. (예: `[FIX] ... (사용자 승인 완료)`)
4. **동시 푸시 수칙 (Dual Push Policy)**: 사용자가 메인 레포지토리의 `git push`를 요청하거나 AI가 직접 push를 수행할 경우, 반드시 중첩되어 연결된 독립 레포지토리인 `intelligence/` 디렉터리 내에서도 동일하게 커밋 및 푸시가 순차적으로 동시 수행되도록 처리합니다. (예: `intelligence/` 내부의 변경 사항을 먼저 커밋/푸시한 후, 메인 레포지토리를 푸시하는 방식 등)
