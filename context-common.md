# context-common.md (AI 에이전트 공통 준수 컨텍스트)

이 문서는 이 프로젝트에서 활약하는 모든 AI 에이전트(Agents)가 안전하고, 예측 가능하며, 일관성 있게 작업을 수행하기 위해 반드시 준수해야 하는 **공통 동작 원칙 및 개발 표준**을 기술합니다.

---

## 1. 절대적 안전 장치 (Safety Lock)

### 1) No-Mutation (기존 소스 변경 금지)
- 사용자의 명시적인 수정 승인 및 직접적인 지시가 내려지기 전까지는 기존에 개발된 비즈니스 소스 코드(`core/`, `pages/`, `service/`, `queries/`, `app.py` 등)를 **단 한 줄도 수정, 가공, 삭제 또는 덮어쓰기 해서는 안 됩니다.**
- 탐색 중 버그가 발견되어 변경이 불가피한 경우에도 먼저 `ai/agent` 영역 밖으로 변경 전 Diff와 리포트를 제출하고 대기해야 합니다.

### 2) Sandbox 영역 (tests/)
- 비즈니스 로직이나 데이터 수집 모듈의 안전성 및 정상 동작 여부를 테스트할 때는 오직 `tests/` 디렉터리 아래에 독립적인 테스트 스페이스 및 임시 코드를 구축해서 검증해야 합니다. 이 과정에서 기존 프로덕션 코드를 mock 패칭하는 것도 메모리 상에서만 일시적으로 수행되어야 합니다.

### 3) 승인 단계별 수정 권한
- **승인 전**: 기존 프로덕션 소스(`core/`, `pages/`, `service/`, `queries/`, `app.py`)는 읽기 전용입니다. 새 검증 코드는 `tests/` 또는 `ai/` 하네스 영역에만 추가합니다.
- **승인 후**: 사용자가 명시한 파일 또는 경로만 수정합니다. 승인 범위를 벗어난 파일에서 추가 문제가 발견되면 별도 Run ID와 리포트로 분리합니다.
- **항상 금지**: `.env`, credential, production infra, 운영 DB write/destructive query, 민감 로그 원문 출력은 수정 승인 여부와 무관하게 금지합니다.
- **프로덕션 코드 수정 시 필수**: `ai/runs/run_*` 아카이브, `diff.patch`, `test.log`, `risk.md`, PR 본문 Run ID, `make verify` 또는 동등한 검증 로그를 남깁니다.

### 4) 컨텍스트 문서 보관 및 명명 규제 (SSOT & Prefix 원칙)
- **컨텍스트 단일 보관**: 모든 시스템/도메인 설명 및 에이전트 준수용 컨텍스트 문서는 **오직 `ai/` 또는 `ai/context/` 폴더 내에만 보관**해야 합니다.
- **명명 Prefix 필수 적용**: 컨텍스트 식별이 항상 직관적으로 가능하도록, 파일명 시작 부분에 반드시 `context-` 접두사를 명시하여 보관해야 합니다. (예: `context-common.md`, `context-queries.md`, `context-service.md`, `context-core.md`, `context-pages.md`)
- **개별 생성 엄금**: `pages/`, `core/`, `service/`, `queries/` 등 실 개발 소스 디렉터리 내에 독립적인 `CONTEXT.md`를 별도로 두는 것은 전면 금지되며, 발견 시 예외 없이 `ai/context/context-<도메인>.md` 형식으로 이관 및 정리해야 합니다.

## 2. 개발 아키텍처 및 코딩 표준 준수

### 1) 파라미터 데이터클래스 규격 통일
- 신규 스킬(`ai/skill/`)을 설계하거나 서비스 레이어 모듈을 결합할 때, 함수의 인자는 개별 파라미터가 아닌 `core/params/parameters.py`에 선언된 데이터클래스(`dataclass`) 구조로 일원화하여 전달해야 합니다.
- 임의의 독립 인자로 함수 시그니처를 변형하는 것은 금지됩니다.

### 2) 서비스 레이어 최우선 경유 및 캐싱 활용
- 에이전트가 특정 도메인(CQMS, GMES, HOPE, HGWS 등)의 품질 지표 데이터나 생산 데이터를 분석하고자 할 때는 원천 쿼리 빌더를 직접 수정하여 호출하기보다는, 이미 완성된 서비스 가공 모듈(`service/*_df.py`)의 캐싱 처리된 API를 경유하여 취득하는 것을 최우선으로 삼습니다.
- 이는 Databricks 및 Oracle 등 원천 데이터베이스에 대한 불필요한 과부하 및 쿼리 비용 발생을 방지하기 위함입니다.

### 3) 순환 참조(Circular Import) 방지
- `ai/agent`, `ai/skill`, `ai/context`, `ai/hook` 모듈 설계 시 각 레이어 간에 순환 참조가 발생하지 않도록 의존성을 명확하게 하향식으로만 배치해야 합니다.
- `ai/skill/`은 `ai/agent/`를 임포트할 수 없으며, `ai/agent/`는 `ai/skill/`과 `ai/context/`를 활용해 업무를 지휘하는 단방향 흐름을 유지해야 합니다.

### 4) 서비스 vs 시각화 전처리 경계 분리 (Separation of Concerns)
- **역할 격리 준수**: 통계 가공 및 비즈니스 룰 처리는 오직 서비스 레이어(`service/df_*.py`)에서만 수행하고, 차트 포맷팅, 마우스 호버(Hover Text) 결합, Top-N 처리 등 시각화 전용 포맷팅은 오직 플롯 레이어(`*_plots.py`)에서만 처리해야 합니다.
- **기준 상세 참조**: 세부 판정 기준 및 예시는 [ai/context/context-preprocessing-boundary.md](file:///home/jumasi/workstation/ai/context/context-preprocessing-boundary.md) 문서를 절대적으로 참조하여 코드 조작을 수행해야 합니다.

## 3. 에이전트 책임 분리 (Separation of Concerns)

에이전트 구현 시 다음 4가지 핵심 서브 패키지의 책임을 엄격하게 분리하여 모듈화합니다.

| 레이어 | 주요 역할 및 준수 규칙 |
| :--- | :--- |
| **`ai/agent/`** | **지휘 및 추론**: 최적의 목표 달성을 위해 계획을 설계하고, 상황에 알맞은 Skill을 선택하며 추론 과정을 실행합니다. 직접적인 DB 쿼리나 UI 제어는 수행하지 않고 오직 오케스트레이션에 집중합니다. |
| **`ai/context/`** | **상태 및 맥락 추적**: 에이전트의 영속 세션, 대화 기억, 분석 결과 기록을 임시 보관하거나 SQLite 및 파일 시스템에 안전하게 저장 및 로딩하는 일을 전담합니다. |
| **`ai/hook/`** | **비즈니스 이벤트 감지**: 에이전트가 깨어날 타이밍(Streamlit 생명주기 진입, 백그라운드 갱신 작업 완료, 임계 품질 지표 초과 시)을 감지하고 에이전트 구동 엔진에 알리는 역할만 수행합니다. |
| **`ai/skill/`** | **도구(Tools/API)**: 에이전트가 손발처럼 사용할 도구 인터페이스입니다. 함수 선언 시 상세한 Docstring과 정확한 타입 선언을 통해 LLM 에이전트가 도구의 목적 및 파라미터를 자율적으로 추론할 수 있게 설계합니다. |

## 4. 예외 처리 및 로깅 원칙 (Error Handling & Traceability)

- **상위 예외 전파**: 에이전트 런타임 혹은 도구 실행 도중 에러가 발생한 경우 에러를 임의로 묵인(`except: pass`)해서는 안 됩니다. 명확한 에러 트레이스백과 문맥 정보를 에이전트 세션 컨텍스트에 정확히 보존하고 상위 엔진으로 던져야 합니다.
- **감사 및 추적성 제공**: 에이전트가 어떤 데이터 스킬을 썼고, 어떤 결과값을 도출했는지 추적할 수 있도록 주요 액션 단계마다 `ops.db` 또는 전용 `log.db`를 경유해 로깅을 남기거나 로컬 로그 파일에 타임스탬프와 함께 정교히 기록해야 합니다.

## 5. Git 커밋 메시지 작성 규칙 (Global Korean Commit Rule)

이 프로젝트의 모든 사람 개발자 및 AI 에이전트는 작업을 성공적으로 완수하고 Git 커밋을 제출할 때, **VS Code 내장 Git 및 GitLens 등의 도구를 포함하여 반드시 다음 한국어 기반 규격**을 준수해야 합니다. 영문 전용 커밋 메시지는 커밋이 거부되도록 엄격히 통제됩니다.

### 1) 구조 준수 및 한국어 작성 원칙
- **언어**: 커밋 제목, 세부 사항 설명은 **오직 한국어**로만 서술합니다.
- **포맷**: 루트에 마련된 `.gitmessage` 템플릿 형식을 철저히 따릅니다.
  ```
  <타입>: <한국어 요약 제목>

  <본문 내용 (선택)>
  ```

### 2) 타입별 프리픽스 규칙 (반드시 소문자 준수)
- `feat`     : 신규 기능, 스킬, 훅, 혹은 페이지 추가
- `fix`      : 에러, 버그 해결
- `docs`     : 컨텍스트 및 AI 에이전트 가이드 문서 생성/수정
- `style`    : 코드 포맷팅 수정 (동작에 무관)
- `refactor` : 코드 리팩토링 및 구조 개선
- `test`     : 테스트 케이스 추가 및 수정
- `chore`    : 패키지 변경, 단순 파일 무시 추가 등 빌드/설정 작업

### 3) 작성 태도 및 양식 준수
- 제목 끝에 온점(`.`)을 찍지 마십시오.
- 현재형 명사형 종결 어미(예: `~ 추가`, `~ 수정`, `~ 보완`, `~ 해결`)로 끝을 맺으십시오.
- 본문을 적을 때는 단순히 "코드를 수정했다"가 아닌 **"무엇을 왜 바꿨는지(What & Why)"**에 대한 명확한 비즈니스 및 아키텍처적 당위성을 기술하십시오.

### 4) VS Code 및 GitLens 도구 규제 설정
VS Code 내장 Git 및 GitLens를 사용할 때도 한국어 커밋 작성을 도모하기 위해 프로젝트 루트의 `.vscode/settings.json` 및 `.gitmessage` 설정을 자동 연동하고 적용합니다.
* **커밋 템플릿 자동화**: VS Code Git 패널 진입 시 `.gitmessage` 템플릿 파일 내용이 입력창에 자동 노출됩니다.
* **GitLens AI 영문 생성 금지**: GitLens AI 기능(`Generate Commit Message`) 사용 시, 영어 작성을 방지하고 한국어로 메시지를 자동 유도 및 작성되도록 커스텀 영문 방지 프롬프트(`gitlens.ai.experimental.customPrompt`)가 빌트인되어 있습니다.

### 5) 로컬 Git Hook을 통한 한글 필수화 강제 규제
만일 커밋 메시지에 한글(가-힣)이 단 한 글자도 포함되어 있지 않으면, 커밋 실행 시 로컬 Git Hook(`commit-msg`) 정책에 의해 자동으로 커밋 실행이 거부(Exit 1)됩니다.
- **활성화 명령어**: 로컬 템플릿 및 Hook 동기화를 위해 개발 참여 및 하네스 개발 진행 전 아래 설정을 터미널에서 반드시 선행 적용합니다.
  ```bash
  # 1. 로컬 템플릿 적용
  git config --local commit.template .gitmessage
  ```
