# [의견서] 시스템 로그 관리를 위한 전용 Agent 도입 타당성 검토 및 로드맵

시스템 운영 과정에서 로그 관리의 중요성을 깊이 인지하시고, 이를 고도화하기 위한 방안으로 **"로그 관리 전용 Agent"** 도입을 검토하시는 것은 매우 선제적이고 훌륭한 접근입니다. 

본 의견서에서는 현재 프로젝트의 **3-Layer 아키텍처 및 Multi-Agent 오케스트레이션 체계**를 바탕으로, 전용 로그 에이전트 도입의 타당성, 장단점, 구체적인 역할(R&R), 그리고 실무적인 단계별 로드맵을 제안해 드립니다.

---

## 1. 핵심 개념 정의: '로깅 시스템(인프라)' vs '로그 에이전트(지능형 주체)'

가장 먼저 짚고 넘어가야 할 점은 **"로그를 기록하고 수집하는 인프라"**와 **"이를 활용하여 자율적인 행동을 취하는 에이전트"**의 역할을 분리하여 바라보는 것입니다.

```
┌───────────────────────────────────────────────┐
│              1. 로깅 시스템 (인프라)          │
│  - Python logging, SQLite 적재, Loki, ELK 등   │
│  - 시스템의 상태 및 이력을 "기록"하고 "저장"     │
└───────────────────────┬───────────────────────┘
                        │ 데이터 공급
┌───────────────────────▼───────────────────────┐
│             2. 로그 관리 에이전트 (AI)        │
│  - 로그 분석, 비정상 감지, 자동 패치 유도 등 │
│  - 로그 데이터를 바탕으로 "판단"하고 "조치"     │
└───────────────────────────────────────────────┘
```

- **로깅 시스템 (Infra)**: 시스템 구동기, DB 클라이언트(`app/core/db/client.py`), 서비스 전처리 단계에서 발생하는 에러, 성능, 이력을 파일이나 DB에 축적하는 기술적 프레임워크입니다.
- **로그 관리 에이전트 (Agent)**: 이렇게 적재된 로그 데이터를 읽어 들여 **비정상 패턴을 탐지하고, 보안 위반을 검역하며, 성능 병목을 진단하여 개발 에이전트들에게 피드백을 주는 '지능형 감사관'**입니다.

따라서 전용 Agent를 고용하기 전에, **신뢰할 수 있는 로깅 프레임워크(인프라)가 먼저 견고하게 구축되어 있는지**를 먼저 확인해야 합니다.

---

## 2. 전용 로그 관리 에이전트 도입의 손익 분석 (Pros & Cons)

만약 로그를 지속해서 분석하고 대응해야 하는 지능형 주체로서 **로그 관리 에이전트 (가칭 `log-sentinel` 또는 `log-operations-manager`)**를 고용할 경우의 이점과 한계는 다음과 같습니다.

### 👍 주요 장점 (Pros)

1. **실시간 자율 디버깅 피드백 루프 완성**
   - 런타임 에러(Runtime Error)나 데이터 정합성 깨짐 로그가 발생했을 때, 에이전트가 에러 로그를 직접 해석하여 `builder-code-reviewer` 및 `builder-query-preprocessor`에게 해결 방안(Patch 제안)을 즉시 전달할 수 있습니다.
2. **보안 감사 및 규정 준수 (Security Guardrail)**
   - 프로젝트 보안 수칙(예: `.env` 환경변수 누출, 오프더레코드 영역인 `intelligence/note/` 무단 접근 등)을 로그 기반으로 상시 검역하여 개발 프로세스 및 운영 환경의 안전을 강제합니다.
3. **성능 병목의 선제적 감지 및 리팩토링 제안**
   - Databricks 쿼리 실행 지연, Streamlit 캐시(`@st.cache_data`) 미적용으로 인한 병목 로그 등을 분석하여 기획 에이전트(`planner-orchestrator`)에게 "XX 화면의 전처리 모듈 리팩토링이 시급함"과 같은 인사이트 리포트를 정기적으로 자동 발행합니다.

### 👎 잠재적 단점 및 리스크 (Cons)

1. **파이프라인 복잡도 및 리소스(비용) 증가**
   - 에이전트가 하나 늘어날 때마다 프롬프트 컨텍스트, 관리 대장(`agents_registry.json`), 실행 주기 관리가 추가로 필요하며, LLM API 호출 토큰 비용이 상승합니다.
2. **인프라 종속성으로 인한 '공회전' 위험**
   - 원천 로깅 인프라가 부실하면, 로그 에이전트는 "로그 파일 포맷이 올바르지 않습니다", "분석할 로그 데이터가 없습니다"라는 예외 처리만 반복하며 시스템 자원을 낭비(공회전)할 우려가 있습니다.

---

## 3. 로그 관리 에이전트 도입 여부에 대한 최종 의견

> [!TIP]
> **"조건부 고용 (단계적 로드맵에 따른 점진적 도입)"을 적극 추천합니다.**

처음부터 완전히 독립된 전용 로그 에이전트를 Registry에 등록하고 상시 가동하는 것은 오버헤드가 큽니다. 따라서, **"기본 로깅 인프라 구축 ➡️ 기존 검증 에이전트 R&R 확장 ➡️ 전용 독자 에이전트 독립"**의 3단계 접근법을 취하는 것이 가장 효율적이고 안전합니다.

---

## 4. 제안하는 3단계 도입 로드맵 (Roadmap)

### 📌 1단계: 로깅 시스템 인프라 구축 및 단일화 (Immediate)
*에이전트 고용 전, 에이전트가 먹고 분석할 '양질의 먹이(로그 데이터)'를 생산하는 단계입니다.*
- **핵심 작업**:
  - `app/core/db/client.py`에 SQL 쿼리 실행 시간 및 실패 로그 남기기.
  - `app/service/` 전처리 함수 호출 시 인풋 파라미터 및 아웃풋 데이터프레임 크기 로깅.
  - 로그 파일 또는 경량 SQLite DB(`local.data/run_logs.db`)를 구축하여 로그 축적 자동화.

### 📌 2단계: 기존 검증 서브에이전트의 R&R 확장 (Short-term)
*신규 에이전트를 추가하는 대신, 기존에 존재하는 에이전트에게 로그 분석 역할을 임시로 부여합니다.*
- **대상 에이전트**: `quality-evaluator` (품질 평가 서브에이전트)
- **수정 방향**:
  - `quality-evaluator`가 단순히 정적 린트/테스트 코드 검증만 하는 것이 아니라, 테스트 구동 시 발생하는 **"런타임 로그 분석 피드백"**도 수행하도록 역할을 일부 확장합니다.
  - 비용과 복잡도를 최소화하면서 로그 분석의 효용성을 검증해 볼 수 있는 가장 현실적인 대안입니다.

### 📌 3단계: 전용 로그 관리 에이전트 (`log-sentinel`) 정식 고용 (Mid-term)
*1~2단계를 거쳐 로그 기반 디버깅/최적화 효과가 검증되었을 때, 이를 전담할 에이전트를 분리 독립시킵니다.*
- **Registry 등록 명세 예시**:
  - **이름**: `log-sentinel`
  - **역할**: 운영 로그 실시간 모니터링, 성능 병목 리포팅, 보안 감사 및 에스컬레이션.
  - **참고 맥락**: `intelligence/infra/logging-specification.md`, `L2-architecture.md`
  - **출력물**: `intelligence/verification/log-audit-report-*.md`
